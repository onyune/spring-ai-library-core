# 🌍 글로벌 피드백 기반 RAG 고도화 및 팀 협업 방안

## 1. 개요
개인화(특정 유저의 취향)가 아닌, 전체 사용자들의 피드백 통계(Global Feedback)를 활용하여 RAG(검색 증강 생성)의 답변 품질을 높이는 방안입니다. 
대중이 긍정적으로 평가한 도서에 가산점을 주어 검색 결과를 보정하고, LLM이 최종 추천을 할 때 피드백 통계를 참고하도록 지시합니다.

---

## 2. 모듈별 상세 R&R (역할과 책임)

### 📱 Telegram / Feedback 모듈 팀 (피드백 데이터 생산/집계)
텔레그램 봇을 통해 수집된 피드백을 누적해 두었다가, Search 모듈이 원할 때 '도서별 성적표(통계)'로 가공하여 넘겨주는 역할을 합니다.

#### 🎯 해야 할 일 (Task)
1. **통계 전송용 DTO 생성**
   - 각 도서가 대중에게 얼마나 좋은 평가를 받았는지 담을 객체입니다.
   ```java
   public record FeedbackStats(
       long totalCount,   // 총 피드백 수 (최소 신뢰도 검증용, 예: 5건 이상인지)
       long goodCount,    // 긍정(👍) 피드백 수
       long badCount,     // 부정(👎) 피드백 수
       double goodRatio   // 긍정 비율 (goodCount / totalCount, 0.0 ~ 1.0)
   ) {}
   ```

2. **도서 다건 통계 조회 API (Internal Service) 구현**
   - Search 모듈이 10~20개의 검색된 도서 ID를 한 번에 넘기면, 해당 도서들의 통계 데이터를 `Map` 형태로 반환하는 기능을 구현합니다. (N+1 쿼리 성능 이슈 방지)
   ```java
   public interface FeedbackInternalService {
       /**
        * 도서 ID 목록을 던지면, 통계 데이터 맵을 반환합니다.
        * 키(Key)는 도서 ID, 값(Value)은 해당 도서의 FeedbackStats 입니다.
        * 통계가 없는 도서는 맵에 포함하지 않거나 null로 처리해도 무방합니다.
        */
       Map<Long, FeedbackStats> getBooksFeedbackStats(List<Long> bookIds);
   }
   ```
   - *참고:* 집계 쿼리를 매번 날리면 DB에 부하가 갈 수 있으므로, 주기적으로 통계를 계산해두는 배치 스케줄러나 캐시, 혹은 Materialized View 사용을 고려해 주시면 더욱 좋습니다.

### 🔍 Search / RAG 모듈 팀 (나의 역할 - 피드백 데이터 소비/적용)
Feedback 모듈이 준 '글로벌 통계'를 바탕으로 순위를 1차 재조정하고, LLM의 프롬프트(Prompt)를 조립하여 똑똑한 답변을 만들어내는 역할을 합니다.

#### 🎯 해야 할 일 (Task)
1. **1차 검색 결과 보정 (Reranking)**
   - 기존의 하이브리드 검색(키워드+벡터) 결과를 가져옵니다.
   - 검색된 도서들의 ID 리스트를 뽑아 `FeedbackInternalService.getBooksFeedbackStats(bookIds)`를 호출해 통계 Map을 받아옵니다.
   - 통계 중 **총 피드백 수(totalCount)가 5건 이상인 도서만** 필터링합니다. (데이터가 적으면 왜곡이 심하므로)
   - 5건 이상인 도서에 대해 `(긍정 - 부정) / 총 피드백` 비율을 구해, 기존 기본 검색 점수(RRF)에 보너스 점수(최대 ±0.5점 등 한도 설정)로 더한 후 리스트를 재정렬합니다.

2. **RAG 프롬프트 컨텍스트에 텍스트 주입**
   - 검색 결과를 텍스트로 변환하여 LLM에게 넘겨줄 때, 통계가 있는 도서 정보 끝에 텍스트를 덧붙여줍니다.
   - *구현 예시:*
   ```java
   if (stats != null && stats.totalCount() >= 5) {
       bookContext.append(String.format(", 피드백: 긍정 %d건/부정 %d건(긍정률 %.0f%%)", 
           stats.goodCount(), stats.badCount(), stats.goodRatio() * 100));
   }
   ```

3. **LLM 시스템 프롬프트(System Prompt) 개선 🌟 (가장 중요)**
   - LLM이 대중의 인기에만 치우쳐 엉뚱한 책을 대답하는 '피드백 편향'에 빠지지 않도록 프롬프트 가이드라인을 명확하게 줍니다.
   - *프롬프트 추가 내용 가이드:*
     - "도서 내용과 사용자 질문의 **관련성(Relevance)을 평가의 최우선 기준**으로 삼으세요."
     - "제공된 피드백 정보는 유용한 참고 자료입니다. 피드백 긍정률이 높으면 '대중이 만족한 검증된 도서'로 간주하세요."
     - "단, **피드백 정보가 없는 도서라도 내용 관련성이 높다면, 신간이거나 아직 리뷰가 적은 것으로 간주하여 차별 없이 추천**에 포함시키세요."

## 4. 통합 작동 시나리오
1. **사용자**: "스프링 기초를 배울 수 있는 책 추천해줘" 
2. **Search 모듈**: 벡터 검색을 통해 1차 후보 도서 10권을 찾음 (A, B, C...)
3. **Search 모듈**: `Feedback 모듈`에게 10권의 ID 리스트 전달하며 통계 요청.
4. **Feedback 모듈**: 집계 후 "A책은 10명 중 9명이 👍, B책은 아직 피드백 0건..." 정보를 Map으로 묶어서 응답.
5. **Search 모듈**: A책에 가산점을 조금 주어 검색 순위를 위로 올리고, 프롬프트용 텍스트 문자열에 "피드백: 긍정 9건..." 문구를 텍스트로 추가.
6. **Search 모듈**: 최종 완성된 프롬프트를 LLM에게 전송 (RAG 수행).
7. **LLM 응답**: "스프링 기초라면 A책을 가장 추천합니다. 내용이 기초부터 잘 설명되어 있을 뿐 아니라, **실제 많은 사용자(긍정 90%)가 유용하다고 검증한 책**입니다..."

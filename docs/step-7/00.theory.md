## 1. AI Agent란 무엇인가?

### 1.1. 기존 AI와 Agent의 차이

**기존 AI (Chatbot)**:
```
사용자: "자바의 정석 책 정보 알려줘"
   ↓
[AI 분석] → 텍스트 응답 생성
   ↓
"죄송하지만 도서관 데이터베이스에 접근할 수 없어요.
 일반적인 정보만 알려드릴게요..."
```

**AI Agent**:
```
사용자: "자바의 정석 책 정보 알려줘"
   ↓
[AI 분석] → 도구(함수) 선택
   ↓
[도서 검색 API 호출] → 실제 데이터 조회
   ↓
"도서관 시스템에서 '자바의 정석'을 찾았습니다!
 저자: 남궁성, ISBN: 9788960777331, 평점: 4.7/5.0"
```

### 1.2. AI Agent의 정의

> **AI Agent** = 사용자의 목표를 달성하기 위해 **자율적으로 도구(Tools)를 선택하고 실행**하여 실제 행동을 수행하는 AI 시스템

**핵심 특징**:
1. **자율성 (Autonomy)**: 스스로 행동 결정
2. **도구 사용 (Tool Use)**: 외부 시스템과 상호작용
3. **목표 지향 (Goal-Oriented)**: 사용자의 의도 달성
4. **상태 유지 (Stateful)**: 맥락을 기억하고 활용

### 1.3. Agent의 구성 요소

```
┌─────────────────────────────────────────────────┐
│                  AI Agent                       │
├─────────────────────────────────────────────────┤
│                                                 │
│  ┌─────────────┐    ┌─────────────┐           │
│  │  LLM Core   │    │   Memory    │           │
│  │  (GPT-4o)   │    │  (Context)  │           │
│  └──────┬──────┘    └──────┬──────┘           │
│         │                  │                    │
│         └──────────┬───────┘                    │
│                    │                            │
│         ┌──────────▼──────────┐                │
│         │  Planning Module    │                │
│         │  (의도 파악, 순서 결정) │              │
│         └──────────┬──────────┘                │
│                    │                            │
│         ┌──────────▼──────────┐                │
│         │  Tool Execution     │                │
│         │  (함수 호출, API 연동)│                │
│         └──────────┬──────────┘                │
│                    │                            │

```

## 2. MCP (Model Context Protocol)란?

### 2.1. MCP의 정의

> **MCP (Model Context Protocol)** = AI 모델이 **외부 도구와 데이터 소스에 접근할 수 있게 해주는 표준 프로토콜**

**기원**: Anthropic이 개발한 오픈 표준 (2024)

### 2.2. MCP가 해결하는 문제

**문제 1: AI 모델의 격리**
```
LLM은 텍스트만 생성 가능
→ 외부 데이터베이스와 상호작용 불가
→ "자바의 정석 책 정보 알려줘" 질문에 실시간 답변 불가
```

**해결**: MCP로 연결
```
LLM + MCP + 도서관 DB/API
→ 실시간 데이터 조회 가능
→ "도서관 시스템에서 '자바의 정석'을 찾았습니다!
   평점 4.7/5.0, 강남도서관에서 대출 가능합니다."
```

**문제 2: 도구 연동의 복잡성**
```
각 AI 모델마다 다른 도구 연동 방식
→ 개발자가 각각 새로 구현해야 함
→ 호환성 문제
```

**해결**: 표준 프로토콜
```
MCP 표준을 따르는 모든 도구
→ 어떤 AI 모델이든 동일하게 연동
→ 재사용성 향상
```

### 2.3. MCP의 작동 원리

```
[Step 1] 사용자 요청
사용자: "자바 책 검색해서 리뷰도 보여줘"
   ↓
[Step 2] AI 분석
LLM: "도서 검색 도구와 리뷰 조회 도구가 필요하네"
   ↓
[Step 3] MCP 메시지 생성
{
  "type": "tool_call",
  "tools": [
    {"name": "searchBooks", "args": {"query": "자바"}},
    {"name": "getReviews", "args": {"bookId": 123}}
  ]
}
   ↓
[Step 4] 도구 실행
searchBooks("자바") → 결과 반환
getReviews(123) → 결과 반환
   ↓
[Step 5] MCP 응답
{
  "type": "tool_result",
  "results": [
    {...},
    {...}
  ]
}
   ↓
[Step 6] AI 최종 응답
"자바 관련 도서 3권을 찾았습니다..."
```

### 2.4. MCP vs Spring AI Function Calling

| 항목 | MCP (표준) | Spring AI Function Calling |
|-----|-----------|---------------------------|
| **범위** | 범용 표준 프로토콜 | Spring 프레임워크 구현체 |
| **구현** | 여러 언어/프레임워크 | Java/Spring 전용 |
| **복잡도** | 낮음 (추상화됨) | 중간 (직접 구현) |
| **유연성** | 높음 (플랫폼 독립) | 중간 (Spring 의존) |
| **교육용** | 추상적 | 구체적 (실습 가능) |

**> 중요: Spring AI Function Calling과 MCP의 관계**

```
┌─────────────────────────────────────────────────────────┐
│                    MCP (표준 프로토콜)                   │
│         "AI가 도구를 사용하는 방법에 대한 표준"           │
└─────────────────────────────────────────────────────────┘
                           │
           ┌───────────────┼───────────────┐
           ▼               ▼               ▼
   [LangChain]    [Spring AI]      [Anthropic SDK]
      Python          Java           TypeScript
   Function        Function        Tool Use
    Calling         Calling
```

**핵심 이해**:
- **MCP**: AI와 도구 연결의 **표준** (개념/원리)
- **Spring AI Function Calling**: MCP를 **Spring Boot에서 구현한 구체적인 방식**

> **💡 초보자를 위한 설명**:
>
> MCP는 "AI가 도구를 사용하는 방법"에 대한 **국제 표준** 같은 것입니다.
>
> Spring AI Function Calling은 이 표준을 **Spring Boot에서 실제로 구현**한 것입니다.
>
> 마치...
> - **HTTP**: 웹 통신 표준 (MCP와 같은 개념)
> - **Spring MVC**: HTTP를 Java/Spring에서 구현한 프레임워크 (Spring AI Function Calling과 같은 개념)
>
> **이 교육에서는 Spring AI Function Calling을 사용하여 MCP의 개념을 실습합니다!**

**이 교육에서 Spring AI Function Calling을 사용하는 이유**:
1. **구체적인 코드**: 실제 Java 코드로 구현하며 학습 가능
2. **Spring Boot 생태계**: 이미 익숙한 Spring Framework와 자연스러운 통합
3. **초급자 친화적**: 복잡한 설정 없이 `@Description` 어노테이션 하나로 구현 가능
4. **실무 적용**: 실제 Spring Boot 프로젝트에서 바로 활용 가능

**실습에서의 구현 방식**:
```java
// MCP의 개념을 Spring AI Function Calling으로 구현

@Description("도서관 시스템에서 도서를 검색합니다")  // MCP의 tool description
public Function<BookSearchRequest, BookSearchResult> searchBooks() {
    return request -> {
        // 도구 실행 로직 (MCP의 tool execution)
        return bookSearchService.searchBooks(request.query());
    };
}
```

이 코드는 **MCP 표준을 Spring AI가 구현한 것**입니다!

## 3. Tools (Function Calling)이란?

### 3.1. Tools의 정의

> **Tools (도구)** = AI 모델이 **외부 세계와 상호작용할 수 있게 해주는 함수 또는 API**

**다른 이름들**:
- **Function Calling**: OpenAI, Google Gemini 용어
- **Tool Use**: Anthropic Claude 용어
- **Function Calling**: Spring AI 용어
- **Tools**: 일반적인 용어

**모두 같은 의미**: AI가 외부 도구를 호출할 수 있는 기능

### 3.2. Tools가 왜 필요한가?

**문제: LLM의 한계**
```
LLM은 텍스트만 생성 가능
→ 데이터베이스 조회 ❌
→ API 호출 ❌
→ 파일 시스템 접근 ❌
→ 실시간 정보 획득 ❌
```

**해결: Tools로 연결**
```
LLM + Tools
→ 데이터베이스 조회 ✅
→ API 호출 ✅
→ 파일 시스템 접근 ✅
→ 실시간 정보 획득 ✅
```

### 3.3. Tools의 작동 원리

```
┌─────────────────────────────────────────────────────────┐
│                    사용자 요청                          │
│         "자바 책 검색하고 리뷰도 보여줘"                │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│                  AI 모델 분석                           │
│         "어떤 도구가 필요할까?"                          │
└─────────────────────────────────────────────────────────┘
                          ↓
         ┌────────────────┼────────────────┐
         ▼                ▼                ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│  Tool 1      │  │  Tool 2      │  │  Tool 3      │
│  searchBooks │  │  getReviews  │  │  checkLoan   │
│  (검색)      │  │  (리뷰)      │  │  (대출)      │
└──────────────┘  └──────────────┘  └──────────────┘
         ↓                ↓                ↓
┌─────────────────────────────────────────────────────────┐
│                   결과 통합                              │
│         "자바 도서 5권, 리뷰 평균 4.5점..."              │
└─────────────────────────────────────────────────────────┘
```

### 3.4. Tools 지원 모델

**중요**: 모든 AI 모델이 Tools를 지원하는 것은 아닙니다!

| 모델 | Tools 지원 | 특징 |
|------|-----------|------|
| **Qwen 2.5** | ✅ 완벽 지원 | 오픈 소스, 한국어 우수, Ollama 사용 가능 |
| **Llama 3.1+** | ✅ 지원 | Meta, 강력한 성능 |
| **Mistral/Mixtral** | ✅ 지원 | 경량화된 강력 모델 |
| **llama3-korean-blossom** | ❌ 미지원 | 한글 특화지만 Tools 미지원 |
| **Gemini 2.0** | ✅ 완벽 지원 | Google, 멀티모달 |
| **GPT-4o** | ✅ 완벽 지원 | OpenAI, 최고 성능 |

**> 왜 일부 모델만 Tools를 지원하는가?**

Tools 지원을 위해서는 AI 모델이 학습 단계에서 **"함수 호출을 위한 특별한 훈련"**을 받아야 합니다:

```
일반 LLM 학습:
텍스트 → 텍스트 생성

Tools 지원 LLM 학습:
텍스트 → 텍스트 + 함수 호출 형식 생성
예: "searchBooks(query='자바')"
```

### 3.5.1. Function Calling의 실제 작동 방식

**Function Calling은 어떻게 작동하는가?**

```
┌─────────────────────────────────────────────────────────┐
│ Step 1: 사용자 요청                                    │
│ "자바 책 3권 추천해줘"                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ Step 2: LLM 분석 (도구 선택 여부 결정)               │
│ "책을 검색하려면 searchBooks 함수를 호출해야겠다"     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ Step 3: Function Call 요청 (JSON 형식)               │
│ {                                                      │
│   "name": "searchBooks",                              │
│   "arguments": {"query": "자바", "limit": 3}        │
│ }                                                      │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ Step 4: 시스템이 함수 실행                             │
│ searchBooks("자바", 3)                              │
│ → [Book1, Book2, Book3]                              │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ Step 5: 결과를 다시 LLM에게 전달                      │
│ "searchBooks 결과: [Book1, Book2, Book3]"             │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│ Step 6: LLM이 최종 답변 생성                         │
│ "자바 관련 도서 3권을 찾았습니다...                  │
│ 1. 토비의 스프링...                                   │
│ 2. 자바의 정석...                                     │
│ 3. Effective Java...                                  │
└─────────────────────────────────────────────────────────┘
```

**Function Call JSON 형식:**
```json
{
  "name": "함수명",
  "arguments": {
    "parameter1": "값1",
    "parameter2": "값2"
  }
}
```

### 3.5.2. Tools를 지원하는 모델의 학습 차이

**일반 LLM vs Tools 지원 LLM:**

| 항목 | 일반 LLM | Tools 지원 LLM |
|------|----------|-----------------|
| 학습 데이터 | 텍스트만 | 텍스트 + 함수 호출 예시 |
| 출력 형식 | 자연어만 | 자연어 + 함수 호출 JSON |
| 예시 | "책을 검색하겠습니다" | `searchBooks(query="책")` |
| 파인튜닝 | ❌ | ✅ (함수명, 파라미터) |

**학습 과정 예시:**
```
[일반 LLM 학습]
입력: "서울 날씨 알려줘"
출력: "죄송하지만 실시간 날씨 정보를 제공할 수 없어요..."

[Tools 지원 LLM 학습]
입력: "서울 날씨 알려줘"
출력: getWeather(city="서울")
       ↓ (시스템이 함수 실행)
입력: getWeather 결과: "18°C, 맑음"
출력: "서울 현재 날씨는 18도이고 맑습니다."
```

### 3.5.3. Spring AI에서의 Function Calling 구현

**Spring AI FunctionToolCallback 패턴:**

```java
// 1. 함수 정의
public BookSearchResult searchBooks(BookSearchRequest request) {
    return bookService.searchBooks(request.query());
}

// 2. FunctionToolCallback 등록
FunctionToolCallback<BookSearchRequest, BookSearchResult> bookSearchTool =
    FunctionToolCallback.builder("BookSearch", this::searchBooks)
        .description("도서관 시스템에서 도서를 검색합니다")
        .inputType(BookSearchRequest.class)
        .build();

// 3. ChatClient에 등록
ChatClient chatClient = ChatClient.builder(chatModel)
    .defaultFunctions(bookSearchTool)  // 또는 defaultToolCallbacks
    .build();

// 4. AI가 자동으로 함수 호출
String response = chatClient.prompt()
    .user("자바 책 추천해줘")
    .call()
    .content();
```

**중요 포인트:**
1. **description**: AI에게 함수의 용도를 설명 (자연어)
2. **inputType**: 함수 파라미터의 Java 타입 (JSON 변환용)
3. **함수 등록**: ChatClient 빌더시 함수 등록

### 3.5.4. Tools 제약 사항 및 주의점

**1. 함수명 명명 규칙:**
```java
✅ 좋은 예:
- searchBooks (동사, 명확)
- getReviews
```

## 5. A2A (Agent-to-Agent) 통신이란?

### 5.1. A2A의 정의

> **A2A (Agent-to-Agent)** = **여러 AI Agent가 서로 통신하며 협력**하여 복잡한 작업을 수행하는 패턴

### 5.2. 단일 Agent vs 다중 Agent

**단일 Agent**:
```
사용자 → [Library Assistant Agent]
            ↓
         [DB/API 직접 호출]
            ↓
         응답 반환
```

**다중 Agent (A2A)**:
```
사용자 → [Coordinator Agent]
            ↓
    ┌───────┼────────┐
    ▼       ▼        ▼
[Search  [Review   [Loan
 Agent]   Agent]   Agent]
    │       │        │
    └───────┴────────┘
            ↓
       결과 통합
            ↓
         응답 반환
```

### 5.3. A2A 통신 예시

**시나리오**: 도서 추천 시스템

```
[사용자 요청]
"초보자를 위한 자바 책 추천해줘"
   ↓
[Coordinator Agent 분석]
"도서 검색 + 리뷰 분석 + 대출 가능 확인 필요"
   ↓
[A2A 통신 #1]
Coordinator → Search Agent
"자바 초보자용 도서 검색해줘"
   ↓
[Search Agent 응답]
"자바의 정석, 이것이 자바다, Head First Java 3권 찾음"
   ↓
[A2A 통신 #2]
Coordinator → Review Agent
"이 3권의 리뷰 분석해줘"
   ↓
[Review Agent 응답]
"자바의 정석 평점 4.7, 이것이 자바다 4.5, Head First 4.8"
   ↓
[A2A 통신 #3]
Coordinator → Loan Agent
"이 책들 대출 가능한 도서관 조회해줘"
   ↓
[Loan Agent 응답]
"강남도서관에서 2권 대출 가능"
   ↓
[최종 응답]
"초보자에게는 Head First Java를 추천합니다.
 강남도서관에서 대출 가능합니다!"
```

### 5.4. A2A 통신의 장단점

**장점**:
- **전문화**: 각 Agent가 특정 분야에 특화
- **확장성**: 새로운 Agent 쉽게 추가
- **유연성**: 작업에 따라 Agent 조합 변경
- **격리**: 한 Agent 장애가 전체 시스템에 영향 적음

**단점**:
- **복잡도**: 통신 프로토콜 관리 필요
- **지연**: Agent 간 통신으로 인한 대기 시간
- **디버깅**: 여러 Agent의 상호작용 추적 어려움

### 5.5. Spring AI에서의 A2A 구현

```java
// Coordinator Agent
@Service
public class CoordinatorAgent {

    private final SearchAgent searchAgent;
    private final ReviewAgent reviewAgent;
    private final LoanAgent loanAgent;

    public String coordinate(String userQuery) {
        // 1. 의도 파악
        Intent intent = analyzeIntent(userQuery);

        // 2. Agent 간 통신
        if (intent.needsSearch()) {
            SearchResult search = searchAgent.search(userQuery);
            // 결과를 다음 Agent로 전달
        }

        if (intent.needsReview()) {
            ReviewResult review = reviewAgent.analyze(search);
            // 결과를 다음 Agent로 전달
        }

        // 3. 결과 통합
        return aggregateResults(search, review, loan);
    }
}
```

### 7.2. MCP 관점에서의 데이터 흐름

```
[1. 사용자 입력]
"자바의 정석 책 정보랑 리뷰도 알려줘"
   ↓
[2. LLM이 MCP 메시지 생성]
{
  "need": ["search", "review"],
  "context": {
    "query": "자바의 정석",
    "intent": "book_info_and_review"
  }
}
   ↓
[3. Function Selection (Spring AI)]
- searchBooks() 호출
   ↓
[4. Tool Execution]
BookSearchService.searchBooks("자바의 정석")
→ Result: {bookId: 123, isbn: "978..."}
   ↓
[5. 다음 Function 결정]
AI: "bookId를 얻었으니 getReviews 호출"
   ↓
[6. Review Function 실행]
ReviewService.getReviews(123)
→ Result: {rating: 4.7, summary: "..."}
   ↓
[7. MCP 응답 생성]
{
  "tool_results": [
    {"tool": "searchBooks", "result": {...}},
    {"tool": "getReviews", "result": {...}}
  ]
}
   ↓
[8. 최종 응답]
"자바의 정석을 찾았습니다!
 평점은 4.7/5.0입니다..."
```

### 7.3. A2A 확장 가능성

**현재**: 단일 Agent (Library Assistant)

**향후 확장**: 다중 Agent 시스템

```
[Main Library Assistant]
         │
    ┌────┼────┬─────────┐
    ▼    ▼    ▼         ▼
[Search] [Review] [Loan]  [Recommendation]
 Agent    Agent   Agent       Agent
                              │
                              ▼
                        [Personalization
                             Agent]
                              │
                              ▼
                        [Feedback
                             Agent]
```

**각 Agent의 역할**:
- **Search Agent**: 도서 검색 전담
- **Review Agent**: 리뷰 분석 및 요약
- **Loan Agent**: 대출 가능 여부 확인
- **Recommendation Agent**: AI 추천 생성
- **Personalization Agent**: 사용자별 개인화
- **Feedback Agent**: 사용자 피드백 수집
